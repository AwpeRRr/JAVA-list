# 开心农场三个版本内容与升级分析

## 一、分析对象总览

本次分析的三个项目版本如下：

| 版本 | 项目目录 | 核心定位 |
| --- | --- | --- |
| 第一版 | `HappyFarmTest` | 使用固定长度数组保存农场对象，完成基础增删查、照料和类型判断 |
| 第二版 | `HappyFarmTest_ArrayList` | 使用二维 `ArrayList` 保存农场对象，引入农场行和位置概念 |
| 第三版 | `HappyFarmTest_ArrayList_4th` | 在二维 `ArrayList` 版本上增加文件保存和读取，实现数据持久化 |

三个版本都围绕同一套面向对象模型展开：`FarmObject` 是抽象父类，`Wheat`、`Corn`、`Chicken`、`Cow` 是具体子类，每个子类通过重写 `care()` 方法体现多态。

---

## 二、第一版 `HappyFarmTest` 内容分析

### 1. 项目结构

第一版主要包含 6 个 Java 源码文件：

| 文件 | 作用 |
| --- | --- |
| `FarmObject.java` | 抽象父类，保存编号、名称、类型，定义抽象照料方法 `care()` |
| `Wheat.java` | 小麦类，继承 `FarmObject`，实现小麦的照料行为 |
| `Corn.java` | 玉米类，继承 `FarmObject`，实现玉米的照料行为 |
| `Chicken.java` | 鸡类，继承 `FarmObject`，实现鸡的照料行为 |
| `Cow.java` | 牛类，继承 `FarmObject`，实现牛的照料行为 |
| `FarmTest.java` | 主程序入口，负责菜单、输入、数组存储和所有业务操作 |

### 2. 核心设计

第一版的核心数据结构是固定长度数组：

```java
private static final int MAX_CAPACITY = 10;
private static FarmObject[] farm = new FarmObject[MAX_CAPACITY];
private static int count = 0;
```

数组 `farm` 保存所有农场对象，`count` 表示当前已经保存的对象数量。数组容量固定为 10，因此第一版农场最多只能保存 10 个对象。

`FarmObject` 中使用静态变量 `nextId` 自动生成编号：

```java
private static int nextId = 1;
this.id = nextId++;
```

因此用户创建对象时只需要输入名称和类型，不需要手动输入编号。

### 3. 功能内容

第一版菜单功能较完整，主要包括：

1. 创建农场对象。
2. 根据名称查找农场对象。
3. 根据类型关键字输出农场对象。
4. 输出指定编号之前的农场对象。
5. 根据编号照料指定农场对象。
6. 判断指定农场对象的具体类型。
7. 根据编号收获或移除农场对象。
8. 输出全部农场对象。

这一版已经体现了 Java 面向对象的几个关键知识点：

- 抽象类：`FarmObject` 定义公共属性和抽象行为。
- 继承：四个具体对象类继承父类。
- 方法重写：不同对象有不同的 `care()` 实现。
- 多态：通过 `FarmObject obj` 调用 `obj.care()`，运行时执行具体子类的方法。
- `instanceof`：通过 `checkInstanceType()` 判断对象真实类型。

### 4. 第一版优点

第一版的优点是简单、直观，适合作为入门版本。

它把农场对象抽象成统一父类，用子类表示小麦、玉米、鸡、牛，能够清楚展示继承、多态和抽象方法。固定数组结构也比较容易理解，适合初学阶段掌握对象数组的使用方式。

功能上，第一版并不只是简单添加和输出，还包含按名称查找、按类型输出、按编号范围输出、按编号删除、照料对象、判断对象真实类型等操作，覆盖了较多基础练习点。

自动编号设计也比较友好。用户不用手动管理编号，减少了编号重复、输入错误等问题。

### 5. 第一版缺点

第一版最大的问题是数据结构限制明显。数组容量固定为 10，超过容量后无法继续添加，扩容需要手动重写逻辑。

第二个问题是 `FarmTest` 承担了太多职责。它既负责用户交互，又负责数据存储，还负责查找、删除、输出等业务逻辑，类的职责不够清晰。

第三个问题是数组删除需要手动移动元素：

```java
for (int i = indexToRemove; i < count - 1; i++) {
    farm[i] = farm[i + 1];
}
farm[count - 1] = null;
count--;
```

这种写法能训练数组操作，但代码容易出错，维护成本也比集合类高。

此外，第一版没有持久化能力，程序退出后所有数据都会丢失；输入校验也不够完整，例如名称没有去空格和空字符串判断。

## 三、第二版 `HappyFarmTest_ArrayList` 内容分析

### 1. 项目结构

第二版在第一版基础上新增了 `Farm.java`，结构变为：

| 文件 | 作用 |
| --- | --- |
| `FarmObject.java` | 抽象父类，保存编号、名称、类型，定义 `care()` |
| `Wheat.java`、`Corn.java`、`Chicken.java`、`Cow.java` | 四个具体农场对象类 |
| `Farm.java` | 新增农场管理类，封装二维 `ArrayList` 和核心操作 |
| `FarmTest.java` | 主程序入口，主要负责菜单和用户输入 |

### 2. 核心设计

第二版最重要的变化是把固定数组升级为二维 `ArrayList`：

```java
private ArrayList<ArrayList<FarmObject>> farmRows = new ArrayList<ArrayList<FarmObject>>();
```

外层 `ArrayList` 表示农场的行，内层 `ArrayList` 表示某一行中的对象。

例如：

```text
第 0 行：小麦、鸡
第 1 行：牛
```

在程序中就可以表示为：

```text
farmRows.get(0) -> [Wheat, Chicken]
farmRows.get(1) -> [Cow]
```

第二版不再使用第一版 `FarmObject` 中的静态自增编号，而是让用户手动输入编号：

```java
public FarmObject(int id, String name, String type)
```

`Farm` 类通过 `hasId(int id)` 检查编号是否重复。

### 3. 功能内容

第二版菜单功能调整为：

1. 重新初始化农场行数。
2. 创建并添加农场对象。
3. 输出所有农场对象和对象总数。
4. 根据名称查找农场对象。
5. 照料指定位置的农场对象。
6. 删除指定位置的农场对象。
7. 清空农场对象。

和第一版相比，第二版更强调“农场空间结构”。对象不再只是简单排在一个数组里，而是属于某一行、某一个位置。

### 4. `Farm` 类承担的核心工作

`Farm.java` 中集中管理农场数据，主要方法包括：

| 方法 | 功能 |
| --- | --- |
| `initFarm(int rowCount)` | 初始化指定行数的农场 |
| `getRowCount()` | 获取农场行数 |
| `getRowSize(int rowIndex)` | 获取某一行对象数量 |
| `isValidRowIndex(int rowIndex)` | 判断行号是否合法 |
| `isValidPosition(int rowIndex, int positionIndex)` | 判断行号和位置是否合法 |
| `hasId(int id)` | 判断编号是否已存在 |
| `addObject(FarmObject obj, int rowIndex)` | 向指定行添加对象 |
| `getObject(int rowIndex, int positionIndex)` | 获取指定位置对象 |
| `removeObject(int rowIndex, int positionIndex)` | 删除指定位置对象 |
| `clearFarm()` | 清空所有行中的对象 |
| `getTotalCount()` | 统计对象总数 |
| `printAllObjects()` | 按行输出所有对象 |
| `searchByName(String name)` | 按名称查找对象并显示位置 |

这个变化让数据管理从 `FarmTest` 中分离出来，程序结构比第一版更清晰。

### 5. 第二版优点

第二版最大的优点是使用集合类替代固定数组，解决了容量固定的问题。每一行中的对象数量可以动态增长，删除对象时也不需要手动移动数组元素，直接调用 `remove()` 即可。

新增 `Farm` 类后，代码职责更清楚：`Farm` 管理数据，`FarmTest` 负责菜单交互，虽然还没有完全分层，但比第一版已经明显改进。

第二版还引入了二维结构，让“农场”更像一个有行和位置的空间。输出时能显示每个对象位于第几行、第几个位置，查找结果也更具体。

输入校验也比第一版更完善。例如：

- 农场行数必须大于 0。
- 对象编号必须为正数。
- 编号不能重复。
- 对象名称不能为空。
- 行号和位置必须合法。

### 6. 第二版缺点

第二版为了引入行和位置，减少或取消了一些第一版已有功能，例如按类型输出、输出指定编号之前对象、按编号判断具体类型、按编号删除等。

这并不是语法错误，但从功能完整度看，第二版在“空间结构”上增强了，在“查询维度”上有所减少。

第二版让用户手动输入编号，灵活性变高，但也增加了操作负担。虽然有重复检查，但用户仍然需要自己记住或设计编号。

此外，第二版仍然没有持久化能力。程序退出后，二维 `ArrayList` 中的数据仍然会全部丢失。

`Farm` 类中仍然直接使用 `System.out.println()` 输出结果，说明业务类和显示逻辑还没有完全分离。如果后期要改成图形界面或文件输出，仍需要调整 `Farm` 类。

## 四、第三版 `HappyFarmTest_ArrayList_4th` 内容分析

### 1. 项目结构

第三版延续第二版结构，仍然包含：

| 文件 | 作用 |
| --- | --- |
| `FarmObject.java` | 抽象父类 |
| `Wheat.java`、`Corn.java`、`Chicken.java`、`Cow.java` | 四个具体农场对象类 |
| `Farm.java` | 农场管理类，新增文件保存和读取 |
| `FarmTest.java` | 主程序入口，新增启动读取、菜单保存、退出前保存 |

第三版对象继承结构基本没有改变，主要升级集中在 `Farm.java` 和 `FarmTest.java`。

### 2. 核心设计

第三版仍然使用二维 `ArrayList` 保存运行时数据：

```java
ArrayList<ArrayList<FarmObject>> farmRows;
```

新增的核心能力是把当前农场状态保存到本地文本文件 `farm.txt`，并在下次启动时读取该文件恢复农场。

保存文件格式为：

```text
ROWS    行数
OBJECT  行号    类名    编号    名称
```

示例：

```text
ROWS	2
OBJECT	0	Wheat	101	Wheat-A
OBJECT	1	Cow	202	Cow-B
```

保存类名是一个关键设计，因为读取文件时不仅要恢复对象的属性，还要恢复对象的真实子类类型。只有恢复成 `Cow`、`Wheat` 等具体对象，后续调用 `care()` 时才会继续体现多态。

### 3. `Farm.java` 中新增的功能

第三版在 `Farm.java` 中新增了文件读写相关导入：

```java
import java.io.BufferedReader;
import java.io.BufferedWriter;
import java.io.IOException;
import java.nio.charset.StandardCharsets;
import java.nio.file.Files;
import java.nio.file.Path;
import java.nio.file.Paths;
import java.util.HashSet;
import java.util.Set;
```

新增的核心方法包括：

| 方法 | 功能 |
| --- | --- |
| `saveToFile(String fileName)` | 将农场行数和对象信息保存到文件 |
| `loadFromFile(String fileName)` | 从文件读取并恢复农场状态 |
| `createObjectByClassName(...)` | 根据文件中的类名重新创建具体子类对象 |
| `escapeText(String text)` | 保存前转义名称中的特殊字符 |
| `unescapeText(String text, int lineNumber)` | 读取时恢复被转义的特殊字符 |
| `parseInteger(...)` | 读取文件时安全解析整数并给出错误提示 |

`loadFromFile()` 中还加入了较完整的文件内容校验：

- 存档首行必须是 `ROWS`。
- 农场行数必须大于 0。
- 对象记录必须是 `OBJECT` 格式。
- 行号必须在农场范围内。
- 编号必须为正数。
- 编号不能重复。
- 名称不能为空。
- 类名必须是 `Wheat`、`Corn`、`Chicken`、`Cow` 之一。

读取时先把数据放入临时集合 `loadedRows`，全部读取成功后才替换当前农场：

```java
farmRows.clear();
farmRows.addAll(loadedRows);
```

这种做法比边读边直接改当前农场更稳妥。如果文件中间出现错误，不会把农场改成一个只读取了一半的不完整状态。

### 4. `FarmTest.java` 中新增的功能

第三版在 `FarmTest.java` 中新增保存文件常量：

```java
private static final String SAVE_FILE = "farm.txt";
```

程序启动时先尝试读取存档：

```java
if (!loadSavedFarm()) {
    initFarm(scanner);
}
```

菜单中新增保存选项：

```text
8. 保存当前农场状态到 farm.txt
```

退出时新增保存询问：

```text
退出前是否保存当前农场状态到 farm.txt？(y/n):
```

因此第三版的使用流程更加完整：用户可以创建、查看、照料、删除、清空对象，也可以手动保存；下次启动时，程序可以自动恢复上一次保存的农场状态。

### 5. 第三版优点

第三版最大的优点是实现了持久化。第二版的数据只存在内存中，程序退出即丢失；第三版通过 `farm.txt` 保存数据，让农场状态可以跨多次运行保留。

第三版不仅保存对象编号和名称，还保存对象真实类名，能够恢复正确的子类对象。这一点很重要，因为它保证了多态行为不会丢失。

文件读写使用 `StandardCharsets.UTF_8`，设计上考虑到了中文名称保存问题。名称保存前还会转义反斜杠、制表符、回车和换行，避免用户输入特殊字符破坏文本格式。

读取文件时的校验也比较完整，能处理存档不存在、格式错误、行号越界、编号重复、对象类型未知等情况，比简单文件读写更可靠。

### 6. 第三版缺点

第三版仍然存在一些可以继续优化的地方。

第一，保存文件名固定为 `farm.txt`，而且是相对路径。程序从不同工作目录启动时，`farm.txt` 的实际位置可能不同，容易让用户找不到存档文件。

第二，文件格式和 Java 类名绑定较紧。比如文件中保存的是 `Cow`，如果以后类名改成 `DairyCow`，旧存档就无法直接读取。更稳妥的方式可以是保存稳定的类型编码，例如 `COW`、`WHEAT`。

第三，仍然需要用户手动输入编号，编号管理没有回到第一版的自动生成方式。对普通用户来说，自动编号会更方便。

第四，程序没有记录“是否修改过但未保存”的状态。退出时会询问是否保存，但如果用户已经保存过且没有再修改，仍然会询问；如果用户清空或重置后忘记保存，也只能靠退出提示提醒。

第五，`Farm` 类中仍然包含大量输出语句，说明业务逻辑和显示逻辑还没有完全解耦。后续如果要做 GUI、Web 或单元测试，会不太方便。

## 五、逐次升级更新点

### 1. 第一版升级到第二版：从固定数组到二维 `ArrayList`

| 对比维度 | 第一版 `HappyFarmTest` | 第二版 `HappyFarmTest_ArrayList` | 升级意义 |
| --- | --- | --- | --- |
| 数据结构 | `FarmObject[]` 固定数组 | `ArrayList<ArrayList<FarmObject>>` | 从固定容量升级为动态集合，并支持二维农场结构 |
| 容量限制 | 最多 10 个对象 | 每行可动态添加对象 | 取消固定容量限制 |
| 农场模型 | 没有独立 `Farm` 类 | 新增 `Farm` 类 | 数据管理从测试类中抽离 |
| 对象位置 | 只有数组下标，用户主要按编号操作 | 明确有行号和位置号 | 农场结构更具体 |
| 编号生成 | `FarmObject` 静态变量自动生成 | 用户手动输入编号 | 编号更可控，但用户负担增加 |
| 添加对象 | 添加到数组末尾 | 添加到指定行 | 可以控制对象所在行 |
| 删除对象 | 按编号删除，手动移动数组元素 | 按行号和位置删除，调用 `ArrayList.remove()` | 删除逻辑更简单 |
| 输出对象 | 输出一维列表 | 按行输出，并显示总数 | 输出更符合二维农场结构 |
| 输入校验 | 有部分数字校验 | 增加正整数、空名称、重复编号、行号位置校验 | 健壮性提高 |
| 功能取舍 | 包含按类型输出、编号范围输出、类型判断 | 重点改为行、位置、清空农场 | 查询功能减少，空间管理增强 |

这一轮升级的核心不是增加保存功能，而是改变数据组织方式。第一版像一个简单对象列表，第二版像一个有多行区域的农场。

从面向对象角度看，第二版新增 `Farm` 类是很重要的进步。第一版中 `FarmTest` 同时负责菜单、数组、查找、删除、输出，职责偏重；第二版把农场数据和操作放进 `Farm` 类，结构更接近真实项目。

不过，第二版也不是单纯“全面增强”。它保留了创建、输出、查找、照料、删除等主流程，但删减了第一版的一些查询功能。因此它更像是一次“数据结构和程序结构升级”，不是所有功能都完整向前兼容。

### 2. 第二版升级到第三版：从内存管理到文件持久化

| 对比维度 | 第二版 `HappyFarmTest_ArrayList` | 第三版 `HappyFarmTest_ArrayList_4th` | 升级意义 |
| --- | --- | --- | --- |
| 数据存储 | 只保存在内存二维 `ArrayList` 中 | 内存二维 `ArrayList` + 本地 `farm.txt` | 程序退出后数据不再必然丢失 |
| 启动流程 | 每次启动都要求初始化农场行数 | 启动时优先读取存档，失败才初始化新农场 | 支持恢复上次状态 |
| 菜单功能 | 没有保存选项 | 新增菜单 `8` 保存当前农场 | 用户可以主动保存 |
| 退出流程 | 直接退出 | 退出前询问是否保存 | 降低忘记保存的风险 |
| 文件格式 | 无 | `ROWS` + 多行 `OBJECT` | 明确记录行数、行号、类型、编号、名称 |
| 类型恢复 | 无需恢复 | 根据类名创建具体子类 | 保存后仍保留多态能力 |
| 异常处理 | 主要处理用户输入错误 | 增加文件不存在、格式错误、重复编号等处理 | 程序健壮性增强 |
| 特殊字符 | 不涉及文件格式 | 名称保存时转义特殊字符 | 避免制表符、换行破坏存档 |
| 数据替换策略 | 不涉及 | 先读入临时集合，成功后整体替换 | 避免半成功读取污染当前数据 |

这一轮升级的核心是持久化。第二版解决了运行期间怎样组织农场对象的问题，第三版解决了程序关闭后怎样保留农场对象的问题。

第三版没有大改对象继承体系，也没有改变二维 `ArrayList` 的主结构，而是在原有基础上增加文件读写。这种升级思路比较稳：先保持已有设计不变，再围绕痛点增加能力。

## 六、三个版本横向对比

| 维度 | 第一版 | 第二版 | 第三版 |
| --- | --- | --- | --- |

## 七、总体演进结论

这三个版本体现了一条比较清楚的升级路线：

```text
第一版：能管理对象
第二版：能用更合理的数据结构管理对象
第三版：能在程序多次运行之间保留对象
```

第一版重点是完成基本功能和面向对象模型；第二版重点是数据结构升级和职责拆分；第三版重点是文件持久化和异常处理。

如果答辩时需要概括，可以这样说：

> 第一版使用对象数组实现基础开心农场，主要练习继承、多态和数组增删查。第二版把数组升级为二维 `ArrayList`，新增 `Farm` 类封装农场数据，支持按行和位置管理对象。第三版在第二版基础上加入 `farm.txt` 文件保存和读取，启动时自动恢复存档，保存时记录对象真实类名，从而保证恢复后的对象仍然具有正确的多态行为。

## 八、后续可优化方向

如果继续升级，可以优先考虑以下方向：

1. 恢复或增强第一版中的查询能力，例如按类型查询、按编号查询、按编号删除。
2. 把编号改为自动生成，或者提供“自动编号 + 手动编号”两种模式。
3. 将 `Farm` 中的输出语句拆出去，让 `Farm` 只返回数据，显示逻辑由 `FarmTest` 负责。
4. 把 `farm.txt` 路径改为可配置路径，或者固定保存到项目目录下。
5. 用稳定类型编码代替 Java 类名，例如保存 `WHEAT`、`CORN`、`CHICKEN`、`COW`。
6. 增加修改对象名称、移动对象位置、按行统计类型数量等功能。
7. 增加单元测试，尤其测试文件读取时的错误格式、重复编号、未知类型等情况。
