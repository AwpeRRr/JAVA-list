# 重要部分代码实现

       我在实现“农场存储结构升级”功能时，主要是把原来一维保存对象的方式改成了二维集合结构。在最早的 `HappyFarmTest` 项目中，农场对象主要保存在 `FarmTest.java` 里的 `FarmObject[] farm` 数组中，并通过 `count` 变量记录当前对象数量。到了 `HappyFarmTest_ArrayList_4th` 项目，我的程序在 `Farm.java` 中使用 `ArrayList<ArrayList<FarmObject>> farmRows` 来保存农场数据。外层 `ArrayList` 表示农场有多少行，内层 `ArrayList<FarmObject>` 表示某一行中具体存放了哪些对象。这样就不再像数组版本那样只能线性保存对象，而是可以通过 `rowIndex` 和 `positionIndex` 两个下标共同定位对象。这一部分代码主要对应 `HappyFarmTest/FarmTest.java` 中的数组成员，以及 `HappyFarmTest_ArrayList_4th/Farm.java` 中的 `farmRows` 成员变量。

       我在实现“农场管理类封装”功能时，主要是把原来集中在测试类中的数据管理逻辑拆分到了 `Farm` 类中。第一版 `HappyFarmTest` 的 `FarmTest.java` 不仅负责菜单交互，还直接负责数组存储、对象创建、查找、删除和输出，类的职责比较集中。第 4 版中，`FarmTest` 主要负责读取用户输入和显示菜单，真正的农场数据操作则交给 `Farm.java` 完成。我认为这样写的好处是结构更清楚，菜单交互和数据管理不会混在一起，也更符合面向对象中职责分离的思想。这一部分代码主要对应 `HappyFarmTest_ArrayList_4th/Farm.java` 中的 `Farm` 类，以及 `HappyFarmTest_ArrayList_4th/FarmTest.java` 中调用 `Farm` 方法的位置。

       我在实现“初始化农场行数”功能时，主要是把农场整体看成一个二维集合结构。在 `Farm` 类中，程序通过 `initFarm` 方法完成初始化。我的处理方式是先清空原来的农场结构，然后根据用户输入的行数循环创建新的空行，每一行都是一个新的 `ArrayList<FarmObject>`。这样重新初始化时不会残留旧数据，也能够准确创建题目要求的指定行数农场。相比最早的一维数组版本，这里新增了“农场行”的概念；相比 `HappyFarmTest_ArrayList`，第 4 版继续保留了这一二维结构设计。这一部分代码主要对应 `HappyFarmTest_ArrayList_4th/Farm.java` 中的 `initFarm` 方法，以及 `FarmTest.java` 中初始化农场的调用流程。

       我在实现“创建农场对象”功能时，仍然保留了父类引用统一管理子类对象的写法。无论用户选择创建小麦、玉米、鸡还是牛，程序最终都会把具体子类对象赋值给 `FarmObject` 类型的引用。这样做利用了向上转型的思想，虽然实际创建出来的是 `Wheat`、`Corn`、`Chicken` 或 `Cow`，但它们都可以作为 `FarmObject` 被统一管理，并存入二维 `ArrayList` 的某一行中。相比第一版，第四版对象编号不再由父类中的静态变量自动生成，而是由用户手动输入，这就让编号更可控，但也要求程序增加编号校验。这一部分代码主要对应 `HappyFarmTest/FarmObject.java` 中的自动编号设计，以及 `HappyFarmTest_ArrayList_4th/FarmObject.java`、`FarmTest.java` 中对象创建和构造方法调用的位置。

       在添加对象功能中，我没有直接把对象随便加入集合，而是先让用户输入对象编号、名称和目标行号。程序会先通过 `farm.hasId(id)` 判断编号是否已经存在，再通过 `farm.isValidRowIndex(rowIndex)` 判断用户输入的行号是否合法。这样可以防止不同对象使用相同编号，也可以避免向不存在的农场行中添加对象。如果编号唯一、名称不为空、行号也合法，程序才会创建对应对象并添加到指定行中。这一部分代码主要对应 `HappyFarmTest_ArrayList_4th/Farm.java` 中的 `hasId`、`isValidRowIndex`、`addObject` 方法，以及 `HappyFarmTest_ArrayList_4th/FarmTest.java` 中的 `createFarmObject` 方法。

       关于二维 `ArrayList` 的添加过程，我选择在 `Farm` 类中封装 `addObject` 方法。只要传入的对象不为空，并且 `rowIndex` 是合法的，就把对象添加到指定行的末尾。添加成功后，`FarmTest` 再通过该行当前长度得到对象所在的位置编号。这样 `FarmTest` 不需要直接操作底层集合，只负责读取用户输入和调用方法。我认为这种写法让菜单交互和数据管理的职责更加分明。相比第一版中直接使用 `farm[count]` 添加对象的方式，第 4 版的添加过程更符合二维农场结构。这一部分代码主要对应 `HappyFarmTest/FarmTest.java` 中的一维数组添加逻辑，以及 `HappyFarmTest_ArrayList_4th/Farm.java` 中的 `addObject`、`getRowSize` 方法。

       我在实现“输出全部农场对象”功能时，使用了双层循环。外层循环遍历每一行，内层循环遍历这一行里的所有对象，并输出每个对象当前所在的行号和位置编号。因为本项目使用的是二维 `ArrayList`，所以不能只像一维数组那样用一个下标遍历，而是必须同时处理 `rowIndex` 和 `positionIndex`。对象总数统计功能也没有继续使用原数组版本中的 `count` 变量，而是通过遍历 `farmRows` 中的每一行，把每一行的对象数量累加起来。这样对象总数始终来源于集合当前真实保存的数据，即使用户添加、删除或清空对象，也不容易出现计数器和实际数据不一致的问题。这一部分代码主要对应 `HappyFarmTest/FarmTest.java` 中的 `outputAll` 方法，以及 `HappyFarmTest_ArrayList_4th/Farm.java` 中的 `printAllObjects` 和 `getTotalCount` 方法。

       名称查找功能的实现上，我选择遍历整个二维 `ArrayList`，将用户输入的名称与每个对象的名称进行匹配。如果匹配成功，就输出该对象的编号、类型、名称，以及它所在的 `rowIndex` 和 `positionIndex`。相比第一版只在一维数组中查找对象，第 4 版的查找结果更加具体，因为它不仅能告诉用户找到了哪个对象，还能告诉用户这个对象位于农场的哪一行、哪一个位置。这一部分代码主要对应 `HappyFarmTest/FarmTest.java` 中的 `searchByName` 方法，以及 `HappyFarmTest_ArrayList_4th/Farm.java` 中的 `searchByName` 方法。

       我在实现“照料指定农场对象”功能时，充分利用了多态特性。在 `careForObject` 方法中，程序先根据用户输入的行号和位置编号，通过 `farm.getObject(rowIndex, positionIndex)` 获取父类类型的 `FarmObject` 对象，然后直接调用它的 `care` 方法。这样 Java 会在运行时根据对象真实类型进行动态绑定，自动执行对应子类内部重写后的照料逻辑。我认为这部分是本项目中最能体现多态的位置。`FarmTest` 并不需要判断这个对象到底是小麦、玉米、鸡还是牛，只需要把它当成 `FarmObject` 来调用照料方法即可。这一部分代码主要对应 `HappyFarmTest_ArrayList_4th/FarmObject.java` 中的抽象照料方法，`Wheat.java`、`Corn.java`、`Chicken.java`、`Cow.java` 中的重写方法，以及 `FarmTest.java` 中的 `careForObject` 方法。

       关于对象类型的显示，本项目没有在输出时反复使用 `instanceof` 判断对象属于哪一个子类，而是通过 `FarmObject` 中的 `type` 属性保存对象类型。每个子类在构造方法中都会把自己的类型信息传给父类，例如小麦传入小麦相关类型，牛传入牛相关类型。这样在输出对象列表和查找结果时，可以直接通过对象的类型属性显示对象所属类型。相比第一版中通过 `instanceof` 判断对象具体类型的功能，这种写法更适合普通列表输出。这一部分代码主要对应 `HappyFarmTest/FarmTest.java` 中的类型判断方法，以及 `HappyFarmTest_ArrayList_4th/FarmObject.java` 和各个子类构造方法中的类型设置。

       创建和删除对象的时候，我选择继续使用 `ArrayList` 作为容器。这样在定位到要删除的对象位置后，只需要调用内层集合的删除方法即可，不需要像数组版本那样手动编写循环覆盖后面的元素，也不需要手动维护 `count--`。在删除相关方法中，程序会先判断行号和位置编号是否合法。如果位置不存在，就返回空值；如果位置存在，就删除并返回被删除的对象。`FarmTest` 再根据返回值判断删除是否成功，并给用户输出对应提示。我认为这样比数组版本更简洁，也降低了手动移位时出现索引错误的风险。这一部分代码主要对应 `HappyFarmTest/FarmTest.java` 中的删除数组元素逻辑，以及 `HappyFarmTest_ArrayList_4th/Farm.java` 中的 `isValidPosition` 和 `removeObject` 方法。

       我在实现“清空农场对象”功能时，并没有直接清空外层 `farmRows`，而是遍历每一行并清空这一行中的对象。这样做的原因是清空对象不等于删除农场行。用户清空后，原来的农场行结构仍然存在，之后还能继续向这些行中添加新的对象。这个功能和重新初始化不同，重新初始化会重新创建农场行，而清空功能只是删除当前农场中已经保存的对象。这一部分代码主要对应 `HappyFarmTest_ArrayList_4th/Farm.java` 中的 `clearFarm` 方法，以及 `FarmTest.java` 中的清空菜单处理流程。

       在处理用户输入行号和位置编号的场景中，我通过 `isValidRowIndex` 和 `isValidPosition` 两个方法集中进行合法性判断。`rowIndex` 必须大于等于 0 并且小于农场总行数，`positionIndex` 必须大于等于 0 并且小于该行对象数量。这样可以避免直接访问二维 `ArrayList` 时产生下标越界异常。对于需要输入数字的场景，我还在 `FarmTest` 中使用统一的整数读取方法处理输入，把字符串转换成整数时如果出现格式错误，就提示用户重新输入。对于对象编号这种必须为正数的输入，又进一步封装了正整数读取方法。这样输入校验逻辑不会散落在多个菜单功能里，代码也更容易维护。这一部分代码主要对应 `HappyFarmTest_ArrayList_4th/Farm.java` 中的 `isValidRowIndex`、`isValidPosition` 方法，以及 `FarmTest.java` 中的 `readInt` 和 `readPositiveInt` 方法。

       第 4 版相比 `HappyFarmTest_ArrayList` 最重要的新增内容，是实现了农场状态的保存功能。在 `Farm` 类中，程序会把当前农场行数，以及每个对象所在行号、真实类名、编号和名称写入本地文件。这样程序退出后，农场对象不会只停留在内存中，而是可以保存到文件里。这里保存真实类名是一个比较关键的设计，因为后续读取文件时需要根据类型重新创建正确的子类对象，而不是只恢复一个普通父类对象。这一部分代码主要对应 `HappyFarmTest_ArrayList_4th/Farm.java` 中的 `saveToFile` 方法，以及 `FarmTest.java` 中的 `saveFarm` 方法。

       我在实现“启动读取存档”功能时，让程序启动后先尝试读取保存文件。如果文件存在并且内容格式正确，就恢复上一次保存的农场状态；如果文件不存在，或者读取过程中发现内容格式有问题，就提示用户重新初始化农场。读取时，程序不仅要恢复行结构和对象属性，还要检查行号是否越界、编号是否重复、名称是否为空、对象类型是否受支持。这样可以减少错误文件对程序运行的影响。我认为这一部分是第 4 版比第二版更完整的地方，因为它让农场数据具备了跨多次运行保留的能力。这一部分代码主要对应 `HappyFarmTest_ArrayList_4th/Farm.java` 中的 `loadFromFile`、`createObjectByClassName`、`escapeText`、`unescapeText` 方法，以及 `FarmTest.java` 中的 `loadSavedFarm` 方法。

       界面交互方面，我仍然选择使用外层 `while` 大循环配合 `switch-case` 菜单进行选择。第 4 版相比第二版增加了保存菜单项，并且在用户选择退出时，会询问是否需要在退出前保存当前农场状态。只有当用户明确输入退出选项时，程序才结束循环，并在最后关闭输入资源。这样既保留了原来菜单式交互的直观性，也让保存功能更容易被用户使用。这一部分代码主要对应 `HappyFarmTest_ArrayList_4th/FarmTest.java` 中的 `main`、`showMenu`、`saveFarm` 和 `askToSaveBeforeExit` 方法。

       总体来看，`HappyFarmTest_ArrayList_4th` 的关键改进不是简单替换容器，而是围绕二维 `ArrayList` 重新组织了农场对象的管理方式，并在此基础上增加了文件保存和读取功能。`FarmTest` 负责菜单和输入，`Farm` 负责集合数据和文件数据的管理，`FarmObject` 及其子类负责对象属性和多态照料行为。相比 `HappyFarmTest`，第 4 版解决了一维数组容量固定、删除需要手动移位、职责过于集中等问题；相比 `HappyFarmTest_ArrayList`，第 4 版进一步解决了程序退出后数据丢失的问题。我认为这样的实现更符合本次作业对 Java 集合框架、二维 `ArrayList`、多态和文件持久化使用的要求。
